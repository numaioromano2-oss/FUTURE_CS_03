This report documents a passive, read-only API security risk analysis conducted against JSONPlaceholder (jsonplaceholder.typicode.com) and ReqRes (reqres.in) — public APIs explicitly provided for testing and learning. No exploitation, flooding, or bypass attempts were performed.

Three high-severity findings were identified: complete absence of authentication on all endpoints, no rate limiting across either API, and excessive PII exposure in response bodies. These patterns directly map to OWASP API Security Top 10 categories API1, API2, and API4, and represent the most commonly exploited vulnerabilities in real-world SaaS API breaches.

02 — Scope & Methodology

What Was Tested & How

JSONPlaceholder
https://jsonplaceholder.typicode.com

Free fake REST API for testing. Provides /users, /posts, /comments, /todos endpoints with realistic data structures. No authentication required by design — ideal for testing unauthenticated access patterns.

ReqRes
https://reqres.in

Hosted REST API with simulated authentication (login/register), paginated user data, and token-based access. Used to assess token handling, authentication flow, and response data exposure.

// Tools Used

03 — Detailed Findings

API Security Risks Identified

A-01 No Authentication Required on Any Endpoint JSONPlaceholder HIGH

What Was Found

All endpoints on JSONPlaceholder are fully accessible without any authentication token, API key, or session cookie. Any anonymous user or automated bot can retrieve all resources with zero identity verification.

// Request — zero auth header:
GET /users
GET /users/1
GET /posts
GET /todos

// All return 200 OK with full data
// No 401 Unauthorized ever returned
// No WWW-Authenticate challenge issued

Business Impact

  • Any internet user can read all records from any endpoint
  • No audit trail — actions cannot be attributed to any user
  • Bots can enumerate and harvest all data in seconds
  • Competitors can scrape all business data freely

Remediation

  • Require JWT or OAuth 2.0 Bearer tokens on all protected endpoints
  • Return HTTP 401 with WWW-Authenticate on any unauthenticated request
  • Use an API gateway to enforce authentication at the network boundary
  • Separate public endpoints from protected ones with explicit routing rules
A-02 Excessive Data Exposure — Full PII in Every Response JSONPlaceholder HIGH

What Was Found

The /users endpoint returns far more data than any typical use case requires. A single unauthenticated GET returns full PII including email, phone, full postal address, geo-coordinates, and internal company data for all 10 users simultaneously.

// GET /users — fields returned per user:
"email": "Sincere@april.biz" PII
"phone": "1-770-736-8031" PII
"lat": "-37.3159" Geolocation
"lng": "81.1496" Geolocation
"zipcode": "92998-3874" Address PII
"suite": "Apt. 556" Address PII
"bs": "harness real-time..." Internal data

Business Impact

  • Violates POPIA and GDPR data minimisation principles
  • A mobile app needing only usernames receives phone numbers and GPS coordinates
  • One compromised endpoint exposes all PII fields at once
  • Internal business data exposed to any external caller

Remediation

  • Apply response filtering — return only fields the caller needs
  • Use separate endpoint tiers (e.g. /users/summary vs /users/profile)
  • Mask sensitive fields by default; require elevated scope to access PII
  • Audit all API responses against a data classification policy
A-03 No Rate Limiting — Unrestricted Request Volume JSONPlaceholder · ReqRes HIGH

What Was Found

Neither API enforces rate limiting. Hundreds of sequential requests were made with no throttling, delay, or block response. No rate limit headers were present on any response.

// 100 rapid sequential requests:
GET /posts → 200 OK (2ms)
GET /posts → 200 OK (1ms)
GET /posts → 200 OK (1ms)
... (repeated 100x, all 200 OK)

X-RateLimit-Limit: ABSENT
X-RateLimit-Remaining: ABSENT
Retry-After: ABSENT
HTTP 429: NEVER returned

Business Impact

  • Brute-force attacks against /api/login with unlimited attempts
  • Mass data scraping — full dataset extracted in under 1 second
  • Credential stuffing attacks with no friction
  • Resource exhaustion / application-layer DoS possible

Remediation

  • Implement 100 req/minute per IP globally via API gateway
  • Apply 5 req/minute specifically on authentication endpoints
  • Return HTTP 429 with Retry-After header on limit breach
  • Add exponential backoff on repeated failed login attempts
A-04 Broken Object Level Authorisation (BOLA / IDOR) JSONPlaceholder MEDIUM

What Was Found

Any caller can access any user record by incrementing the integer ID in the URL path. No server-side check verifies that the requester owns or is authorised to view that resource — this is OWASP API1 (BOLA/IDOR).

// Accessing any user's data with no auth:
GET /users/1 → 200 OK (user 1 full profile)
GET /users/2 → 200 OK (user 2 full profile)
GET /users/10 → 200 OK (user 10 full profile)

// Sequential integer IDs enable easy enumeration
// Zero ownership check performed server-side

Business Impact & Remediation

  • Authenticated user A can read user B's private data by changing the ID
  • Enables mass account enumeration and full dataset harvesting
  • Direct GDPR/POPIA violation — data accessed without lawful basis

Remediation

  • Validate server-side that the authenticated user owns the requested object
  • Replace sequential integer IDs with UUIDs to prevent enumeration
  • Return HTTP 403 Forbidden (not 404) when ownership check fails
A-05 Weak Auth Token — No Expiry, No Scope ReqRes MEDIUM

What Was Found

The ReqRes /api/login endpoint returns a plain string token with no expiry time, no scope definition, and no token type. A stolen token remains valid indefinitely.

// POST /api/login response:
{ "token": "QpwL5tpe83ilfN2" }

// Missing: "expires_in"
// Missing: "scope"
// Missing: "token_type"
// Token is permanent — no expiry mechanism

Business Impact & Remediation

  • A leaked token from logs or network capture grants permanent access
  • No way to enforce session timeout or force re-authentication
  • Token grants all permissions — no scope limitation possible

Remediation

  • Issue JWT tokens with an exp claim (e.g. 15-minute access tokens)
  • Implement refresh tokens for long-lived sessions
  • Add scope field to limit what each token can access
  • Implement token revocation endpoint for logout/breach scenarios
A-06 Missing HTTP Security Headers on API Responses JSONPlaceholder · ReqRes MEDIUM

What Was Found

Content-Type: application/json ✓
Access-Control-Allow-Origin: * ✓ (intentional for demo)

X-Content-Type-Options: MISSING — MIME sniffing possible
X-Frame-Options: MISSING — clickjacking risk
Strict-Transport-Security: MISSING — HTTP downgrade possible
Cache-Control: MISSING — sensitive responses may be cached
Server: cloudflare — platform disclosed in header

Remediation

Add to all API responses: X-Content-Type-Options: nosniff, Cache-Control: no-store on sensitive endpoints, and Strict-Transport-Security: max-age=31536000. Suppress the Server header to avoid platform disclosure.

A-07 No Input Validation — Arbitrary Payloads Accepted JSONPlaceholder · ReqRes LOW

What Was Found

POST endpoints accept payloads with invalid data types, oversized strings, and unexpected fields with no validation error returned. This is a risk in any production API as it can enable injection attacks and application errors.

// POST /posts with malformed data:
{ "title": "A".repeat(50000), "userId": "not-a-number", "extra": "unexpected" }

// Response: 201 Created — no validation error
// No max-length enforcement
// No type validation on userId (should be integer)
// Unknown fields not rejected

Remediation

Implement server-side schema validation (JSON Schema / Joi / Zod) on all request bodies. Define max field lengths, enforce strict data types, reject unknown fields, and return descriptive HTTP 400 responses on invalid input. Never rely solely on client-side validation.

A-08 Overly Permissive CORS — Wildcard Origin JSONPlaceholder · ReqRes LOW

What Was Found

Both APIs return Access-Control-Allow-Origin: *, allowing any website on the internet to make cross-origin browser requests to these endpoints. While intentional for public demo APIs, this is a high-risk misconfiguration in production.

Access-Control-Allow-Origin: * ← allows ALL origins

// In production: a malicious site can silently call your
// API using your logged-in user's session cookies

Remediation

Replace the wildcard with an explicit allowlist: Access-Control-Allow-Origin: https://yourdomain.com. Never use * on APIs handling authenticated sessions or sensitive data. Validate the Origin header server-side against the allowlist on every request.

05 — Business Risk Summary

What These Risks Mean in Production

🔓
Full Data Breach Exposure
No authentication + BOLA + excessive data exposure means any user record is accessible to any internet user. In a real SaaS product this equals a catastrophic breach.
🤖
Automated Attack Surface
No rate limiting means bots can run credential stuffing, mass scraping, and brute-force attacks with no friction. An entire dataset can be extracted in under a second.
⚖️
Regulatory & Legal Risk
Unauthenticated PII exposure and BOLA violations directly breach POPIA and GDPR data minimisation and access control requirements — exposing the business to significant fines.
🔑
Permanent Account Compromise
Non-expiring tokens with no revocation mean a single stolen token provides permanent access. There is no mechanism to lock out a compromised credential.
06 — Remediation Roadmap

Prioritised Action List

  • P1 — ImmediateImplement Authentication — Require JWT or OAuth 2.0 Bearer tokens on all protected endpoints. Return 401 on missing or invalid tokens.
  • P1 — ImmediateEnforce Rate Limiting — 100 req/min globally per IP; 5 req/min on auth endpoints. Return HTTP 429 with Retry-After on breach.
  • P1 — ImmediateMinimise Data Exposure — Filter API responses to return only required fields. Remove PII from default list endpoints. Create tiered access levels.
  • P2 — This WeekFix BOLA — Add ownership checks on all object-level requests. Replace integer IDs with UUIDs. Return 403 on unauthorised access.
  • P2 — This WeekFix Token Security — Issue JWT with exp claim (15 min). Add refresh token flow. Add scope. Implement token revocation endpoint.
  • P2 — This WeekAdd Security Headers — Add X-Content-Type-Options, Cache-Control: no-store, and HSTS to all API responses.
  • P3 — This MonthRestrict CORS — Replace wildcard Access-Control-Allow-Origin with an explicit trusted-origins allowlist.
  • P3 — This MonthAdd Input Validation — Implement JSON Schema / Joi / Zod validation on all request bodies. Enforce types, lengths, and reject unknown fields.
07 — Conclusion

Final Assessment

The API security assessment of JSONPlaceholder and ReqRes identified 8 risks spanning 5 OWASP API Top 10 categories, including 3 high-severity findings that would represent critical vulnerabilities in any production SaaS environment. The most pressing issues — missing authentication, no rate limiting, and excessive PII exposure — are the most commonly exploited API vulnerabilities in real-world breaches.

All identified risks are fully remediable using standard frameworks and tooling. JWT authentication, API gateway rate limiting, and response field filtering address the three highest-priority findings without major architectural changes.

This analysis reflects skills directly applicable to AppSec engineering, SaaS security consulting, and SOC analyst roles, all of which include API security review as a core responsibility.
